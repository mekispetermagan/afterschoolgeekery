Chasing game
What is this project about?

We're going to put together an extremely simple game where one player follows the mouse pointer, and the other player follows the first player. You can find the result here:

Preparations

Start the session by watching the animation "Scratch is Fun!" together. In it, Scratch sprites demonstrate how Scratch works by building and running a very simple Scratch code:

After you have watched the video and discussed what happened in it, everyone should create a new Scratch project, choose a sprite they like (and feel free to delete the cat) and try to rebuild the code together from memory. The blocks will look somewhat different (the animation is over eight years old, and was created for an earlier version of Scratch), but this shouldn't bother the kids at all:

Step 1: Following the mouse pointer

Discuss what is missing from circular movement to make it look like the way sprites move in a game. Among the many ideas, there will most certainly be one that we should be about controlling the movement. The next question to discuss might be how we control sprites in a game, and among the many answers will be the console controller, screen touch, keyboard, and mouse. Of these options, keyboard and mouse control are the easiest to implement on a laptop; and that's what we're going to do.

The direction of movement will be given by the mouse pointer. If you start looking in the motion blocks, you will find one that says just that. If you replace the "turn" block with this one, you will have the mouse control code in one go. It's also worth slowing down the sprite a bit by making the number in the "move" block smaller, so that the game doesn't feel so rushed:

If you don't want the sprite to stand on its head while moving left, you can find a movement block to prevent this. And if you find the sprite too big, you can safely make it smaller by using the "set size to" block in the purple "look" group. (These properties can be set not only by block, but also in the sprite menu below the stage.)

Finally, so that you don't always have to grab and place the sprite before starting the game, you can place it in the top left corner and use a "go to" block to have it automatically moved there every time you start the game:

Step 3: Touch sensing

Now it's worth briefly discussing what else is missing to make this an exciting game. One of the answers will most certainly be to make something happen if the chaser catches its target. The simplest option is that the game simply ends.

But what does it mean in programming language for one sprite to catch another? Well, it means that their costumes touch each other on the stage. Scratch has a special block for this in the light blue "perception" category. This is set to the mouse pointer by default, but you can switch it with a single move:

Importantly, this block will be assigned to the first sprite, the one we control with the mouse. All the Scratch interpreter has to do is waiting until the touch event occurs:

This could easily be the first time children have encountered a sprite running several codes at the same time. This is not usually a problem, as only we adults are conditioned to perform tasks in a linear, sequential way, but for children multitasking is perfectly natural. But if it does come up, you might as well mention that it's called parallel programming, and it's a very cool and modern thing.

To prevent a later error, it is worth adding an extra second before touch detection starts. Since the game ends with the two sprites touching each other, it is possible that on restart the touch sensing script will be the first of the parallel tasks to be performed, and the game will end immediately before the sprites have taken their places:
