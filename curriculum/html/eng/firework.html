Fireworks project
What is this project about?

This is again a very simple but spectacular project that you can do with beginners and little ones. Although we use very simple techniques throughout, there is an advanced element of sprite cloning. Cloning is a watershed in Scratch: those who know how to use it will produce Scratch projects of a higher level than those who don't.

Two versions of this project have been developed; version A is simpler, and version B is more spectacular. The two versions diverge from step 4 onwards.

Step 4A

In this version, we make the animation more spectacular by not launching the stars one after the other, but continuously, so that there are many of them in the air at the same time. But how can one sprite be in several places at once? Well, that's what cloning is for. We make clones of the star sprite continuously, and each clone is launched and then deleted.

To do this, you need to disassemble the existing code a little. You should be careful with this, as it can easily confuse children. However, such interventions cannot be avoided in any creative process; you can't imagine anything worse than showing children the right solution to a problem from the start and depriving them of the pleasure of thinking together.

When splitting the code, it is important to keep in mind which task belongs to the original sprite and which to the clone. And it turns out that everything that concerns the flight of the star will belong to the clone; the original sprite will have only the repeat block used to create clones.

This is a standard technique, not only in Scratch, but in programming in general, and in fact in practically all creative processes: first experiment on a concrete prototype, then generalize and replicate.

Step 4B

Version "A" is spectacular, but more like a sparkler than a firework. Firework projectiles are used to explode in the air. That's exactly what we're going to do in this version: clones of the star will be shot up into the air, and they scatter around in a circle in all directions.

To do this, it is worth thinking together about what "in all directions" means. For adults, it is natural that a circle has 360 degrees, each of which represents a direction, but you cannot take for granted that kids have learned this at school, and certainly not that they have taken it on board. It's worth getting out the “point towards” block once again, opening it up and exploring together how many different directions the pointer can point in.

And you can also get out, draw or search the web for a traditional clock (unfortunately Scratch doesn't have a clock sprite).

Or you can recall the very first program code we used to walk our sprite around and count how many times you have to turn 15 degrees until you make a full circle.

Actually, it doesn't matter what you use as an illustration; the point is to get the kids involved in a counting game, at the end of which it will turn out that a 360-degree full circle is made up of 24 15-degree segments (or 12 30-degree segments, or 18 20-degree segments; it doesn't matter, let the collective reasoning guide you).

Now it will come as natural that a star that has reached the end of its trajectory will rotate 15 degrees 24 times, launching a clone in each direction.

And the clones are moved, with the same "move" block and repeat as the original star.

If you restart the project, you will see a beautiful sight! :) Not exactly what we expected, as the clones come in a spiral line instead of flying in sync; but that's quite a promising result.

Step 5B

To understand why clones come in a spiral, it's worth thinking about how repetition works in Scratch. For example, compare the two codes below:

Although the two are similar, they do not run in the same way. The first one glides the sprite (which is why we used it in the fireworks animations), while the second one teleports it in one big step. The reason for the difference is that while the blocks underneath each other are executed at the same time (in the same screen refresh), Scratch delays the steps of the iterations (one step per screen refresh). The same happens when creating 24 clones: the Scratch player executes the task in 24 screen refreshes, or about one second. We could make it instantaneous by racking 24 cloning and turning blocks on top of each other, but that would be just ugly...

The solution is a bit hidden in Scratch. There is an option to give a piece of code a separate name and create a block for it. This is a basic tool in advanced Scratch programming; it's more or less equivalent to using functions in adult programming.
