Gliding
What is it about?

Scratch gives you two types of "glide" blocks. One takes the sprite to a random location, the other to the coordinates you specify. Both are very easy to use: every time you refresh the screen, the Scratch player calculates for you where the sprite is moving.

Start a new project, replace Scratchie with a ball and two other sprites of your choice, and choose a background.

Step 1

Let's start with a very simple movement. Select one of the non-ball players, and pick the "glide to random position" from the blue motion blocks: .

When you click on the block, the sprite glides to a new place on the stage. If you click again, it glides to a new location. The gliding is sometimes slower, sometimes faster, as the sprite has to move a greater or lesser distance in the given 1 second. This is how the block works.

Step 2

If you combine the two blocks, from now on, every time you press the space bar, the sprite will travel to a new location.

Step 3

Now let's move the other sprite; for a change, let's move this one on a fixed track. Now we will use the other "glide" block, the one with "x" and "y". You may remember from school that these are coordinates. After all, the Scratch stage is a coordinate system with the point (0,0) in the center (so x is 0 and y is 0), and the x coordinates range from -240 on the left to 240 on the right, and the y coordinates range from -180 at the bottom to 180 at the top.

But fortunately, you don't have to worry much about coordinates in this project. The Scratch developers have taken the precaution of making sure that if you place your sprite anywhere on the stage, its x and y coordinates are automatically included in the movement blocks that have not yet been taken. (Note that the coordinates in the blocks already taken out are not changed; and it's a good idea to stop the project before taking out a block, because automatic coordinate insertion doesn't always work during playback.)

So put your sprite in a place you think is appropriate, and pick a "glide" block with the x-y coordinates. Then drag it to another location, and pick another glide block with the new coordinates. Repeat this as many times as you like:

Comparison

There are some important differences between when key control is implemented with event blocks, as in step 3, and when it is implemented with condition blocks, as in step 4. It is worth taking these in turn.

  • When using an event block, the sprite will move on keystroke even when the project is not running. Such projects cannot therefore be stopped completely. At the beginning of the condition block version, there is a "click on the green flag", so this only works when the project is started.
  • The event block version only allows you to move in four directions, not diagonally. In the condition block version, the sprite can move diagonally.
  • You cannot put another event block on top of the event block version; so you cannot, for example, link the code to the arrival of a message or the creation of a clone. With condition blocks we have more freedom; we can change the "green flag" event on top of the code to whatever we want.
  • In the event block version, the movement of the sprite is a bit jerky at the start and when changing direction. This is because the event block asks the computer's operating system whether a keystroke event has occurred; it is not tuned for game control but for typing, and when typing it is important not to accidentally type two letters with a slightly longer keystroke. The conditional block takes the information from somewhere else, so it doesn't have lagging.

In summary, although the first, event-block version is somewhat simpler, in almost all cases the second, conditional-block version works better.
